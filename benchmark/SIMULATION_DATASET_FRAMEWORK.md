# 回忆录项目 · 多模态仿真测试集构建思路与框架

本文档基于「腾讯老师提供的文本回忆录数据 + 多模态大模型 + 本系统算法」的评估思路，给出**整体框架**与**后续实施步骤**，便于团队讨论与落地。

---

## 一、整体思路回顾

| 阶段                  | 输入                                    | 产出                                  | 说明                                                                                 |
| --------------------- | --------------------------------------- | ------------------------------------- | ------------------------------------------------------------------------------------ |
| **1. 图文对齐** | 腾讯文本回忆录文段                      | 每段对应多张「仿真照片」              | 文→图由多模态/图像生成 API 完成，作为后续**Ground Truth 图片**                |
| **2. 算法跑通** | 上述仿真照片                            | 系统生成的问题 + 多轮对话             | 用现有**多模态理解 + 问题生成 + 对话管理** 跑一遍                              |
| **3. 用户模拟** | 用户画像（从回忆录提炼）+ 系统问题      | 模拟用户的回答（照片+文字的真实对话） | 用**另一个 LLM** 扮演「回忆录主人」，回答系统问题，形成 **仿真对话数据** |
| **4. 评估打分** | (1) 腾讯真实回忆录文本 (2) 仿真对话数据 | Judge LLM 的评分与评语                | **LLM-as-Judge** 从风格与内容相似度对比，给出算法评价指标                      |

---

## 二、数据流与模块框架

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                        多模态仿真测试集构建流水线                               │
└─────────────────────────────────────────────────────────────────────────────┘

  [腾讯文本回忆录]
        │
        ▼
  ┌──────────────┐     文段 → 图像生成 API/多模态大模型
  │ 文本→图像     │  ──────────────────────────────────►  仿真图片 (Ground Truth)
  │ (Text2Image)  │     一段可对应多张图
  └──────────────┘
        │
        │ 仿真图片路径 + 对应原文段
        ▼
  ┌──────────────┐     现有 demo 流程：analyze_image → question_generator
  │ 多模态理解    │  ──────────────────────────────────►  问题列表 + 分析结果
  │ + 问题生成    │     (multimodal_analyzer + dialogue_manager)
  └──────────────┘
        │
        │ 问题 + 当前图片分析
        ▼
  ┌──────────────┐     用户画像(从回忆录提炼) + 当前 Q + 历史 QA
  │ 用户模拟 LLM  │  ──────────────────────────────────►  仿真用户回答
  │ (User Sim)   │     多轮直至达到设定轮数
  └──────────────┘
        │
        │ 每张图： [Q1,A1], [Q2,A2], ...
        ▼
  ┌──────────────┐     输入：(1) 原文回忆录 (2) 仿真对话全文
  │ Judge LLM    │  ──────────────────────────────────►  风格/内容相似度评分 + 评语
  │ (评估标准)   │     Prompt 中明确维度与打分标准
  └──────────────┘
```

- **Ground Truth**：腾讯回忆录文本 + 由其生成的仿真图片（及图文对应关系）。
- **算法输出**：系统根据仿真图片生成的问题 + 用户模拟 LLM 的回答，即「仿真对话数据」。
- **评价标准**：Judge LLM 对比「真实回忆录」与「仿真对话」，从风格与内容上打分。

## 三、数据与目录结构建议

### 3.1 腾讯回忆录原始数据假设

若腾讯提供的是**按文段组织的回忆录**，可先约定一种最小可用的结构，例如：

```json
{
  "memoir_id": "memoir_001",
  "title": "某老师回忆录",
  "segments": [
    {
      "segment_id": "seg_01",
      "text": "1985年夏天，我们一家人在老房子前拍了一张合影……",
      "metadata": { "year_mentioned": "1985", "theme": "家庭合影" }
    }
  ]
}
```

若原始数据是纯文本，可在预处理阶段按段落/按字数切分为 `segments`，再写入上述结构。

### 3.2 仿真数据集目录结构

建议在 `demo` 下单独建一个数据集目录，便于版本管理与评估脚本复用：

```
demo/
  simulation_dataset/           # 仿真测试集根目录
    raw/                        # 原始输入
      memoir_001.json           # 腾讯回忆录（按上面格式或你们约定格式）
    text2image/                 # 阶段1 产出
      memoir_001/
        seg_01/
          prompt.txt            # 用于生成图的文本/提示
          image_0.png
          image_1.png
          mapping.json          # 文段与图片对应关系
    pipeline_output/             # 阶段2+3 产出：算法+用户模拟
      memoir_001/
        seg_01/
          image_0/
            analysis.json       # 多模态分析结果
            questions.json      # 生成的问题列表
            dialogue.json       # 仿真对话 [{"question","answer"}, ...]
    evaluation/                 # 阶段4 产出：Judge 评估结果
      memoir_001/
        seg_01_image_0_judge.json   # 评分 + 评语
    config/
      user_persona_memoir_001.json  # 从回忆录提炼的用户画像（供 User Sim 使用）
```

这样从「原始回忆录 → 图文 → 对话 → 评估」全链路都可按 `memoir_id / segment_id / image_id` 追溯。

## 四、各模块职责与接口

### 4.1 文本 → 图像（Text2Image）

- **输入**：一段回忆录文本 `segment["text"]`，可选 `metadata`（年代、主题等）。
- **输出**：多张图片路径 + 与文段的映射（如 `mapping.json`）。
- **实现方式**（二选一或组合）：
  - **方式 A**：调用带「文生图」能力的 API（如腾讯混元文生图、DALL·E、国内其他多模态/文生图接口），用文段或由 LLM 提炼的「图像描述」生成图。
  - **方式 B**：用多模态大模型（聊天界面或 API）先根据文段生成详细的「图像描述」，再交给文生图 API 生成图。
- **建议**：每段至少生成 1 张图，有条件可 2～3 张，用于覆盖不同子情节，便于后续评估时样本更充分。

### 4.2 多模态理解 + 问题生成（复用现有代码）

- **输入**：`text2image` 产出的某张仿真图片路径。
- **流程**：直接复用现有逻辑：
  - `MultimodalAnalyzer.analyze_image(image_path=...)` → `analysis_result`
  - `DialogueManager.start_dialogue(photo_id, analysis_result)` → 初始问题列表
  - 若需多轮：循环 `add_answer(question, answer)`，其中 `answer` 由下一节的「用户模拟 LLM」提供。
- **输出**：每张图对应一份 `analysis.json`、`questions.json`，以及多轮 `dialogue.json`（见 4.3）。

与现有 `server.py` / `main.py` 的差异仅在于：**输入图片来自 text2image，且回答来自 User Sim 而非真人**；核心逻辑可共用。

### 4.3 用户模拟 LLM（User Simulator）

- **目标**：用「回忆录主人」的口吻和已知信息，对系统提出的问题作答，形成贴近真实照片访谈的对话。
- **输入**：
  - **用户画像**：从腾讯回忆录中提炼（见 4.3.1）。
  - 当前图片的 **分析结果**（可选，用于约束回答不偏离画面）。
  - 当前 **问题** + **历史 QA**（多轮时）。
- **输出**：一条自然语言 **回答**；多轮时循环调用，得到完整 `dialogue.json`。

**4.3.1 用户画像从哪里来**

- 用 **另一个 LLM** 对整篇或按段的回忆录做摘要与结构化，产出「用户画像」JSON，例如：
  - 称呼、年龄层、职业/身份
  - 语言风格（简洁/啰嗦、书面/口语、是否带方言感）
  - 关键人物、地点、年代、事件（可做成简短列表或关键词）
  - 可选：每段对应的「本段要点」，供该段对应图片的问答时参考
- 将该画像写入 `config/user_persona_memoir_001.json`，并在 User Sim 的 system/user prompt 中显式使用（见下）。

**4.3.2 User Sim 的 Prompt 设计要点**

- **System**：你是回忆录中的主人公，严格依据给定画像与回忆录内容回答，不要编造画像外的信息；语气、用词、叙事风格需与回忆录一致。
- **User**：当前问题 + 当前图片的简短描述（或分析要点）+ 已有 QA 历史；可加「若回忆录中有明确对应内容，请优先引用」。
- 输出：仅一条回答，不要重复问题、不要元说明。

这样，(1) 腾讯真实数据 与 (2) 仿真对话数据 在「内容来源」上同根（回忆录），便于 Judge 做内容与风格对比。

### 4.4 Judge LLM（LLM-as-Judge）

- **输入**：
  - **(1) 真实数据**：该文段（及可选前后文）的腾讯回忆录原文。
  - **(2) 仿真数据**：该文段对应的一张或多张图的「仿真对话」拼接成的完整对话文本（或按图分块提供）。
- **输出**：结构化评分 + 简短评语，例如：
  - **内容相似度**（1–5）：仿真对话与回忆录在事实、人物、时间、地点等方面的一致性。
  - **风格相似度**（1–5）：语气、用词、叙事方式与回忆录的接近程度。
  - **整体评价**：一段话总结；可选：改进建议。
- **Prompt 设计建议**：
  - 明确说明两段文本分别来自「真实回忆录」与「基于其生成的照片访谈对话」。
  - 给出 1–5 分的定义（例如 5=高度一致，1=几乎无关），并要求先输出分数再写评语。
  - 要求输出为 JSON，便于后续汇总与画表（如按 memoir/segment/image 聚合求平均分）。

该 Judge 的分数即可作为「算法在仿真测试集上的评价标准」：同一套仿真图片与用户画像下，若你们改进问题生成或对话策略，只需重新跑 4.2+4.3，再重新跑 4.4，比较分数变化。

## 五、后续实施步骤（推荐顺序）

### 阶段 0：数据与约定（优先）

1. **确认腾讯回忆录数据格式**

   - 若已是 JSON/表格，约定 `memoir_id`、`segment_id`、`text`、以及可选 metadata。
   - 若是纯文本，约定分段规则（按段、按字数、按标题），并写一个脚本输出为上述 `raw/memoir_xxx.json`。
2. **建目录与配置**

   - 在 `demo` 下创建 `simulation_dataset/` 及子目录（如 `raw/`, `text2image/`, `pipeline_output/`, `evaluation/`, `config/`）。
   - 可选：在 `config.py` 或单独 `simulation_dataset/config.yaml` 里写「每段最多生成几张图」「对话最大轮数」等，便于调参。

### 阶段 1：文本 → 图像

3. **实现 Text2Image 模块**

   - 输入：`raw/memoir_001.json` 或单段 `text`。
   - 若使用「文段 → LLM 生成图像描述 → 文生图 API」：先实现「文段 → 图像描述」的调用（可用现有 Gemini/混元文本 API），再接文生图 API。
   - 输出：每段对应 1～3 张图，保存到 `text2image/memoir_001/seg_01/`，并写 `mapping.json`（文段 id、图片路径、可选 prompt）。
4. **批量脚本**

   - 对 `raw/` 下所有回忆录遍历段，调用 Text2Image，写满 `text2image/`；注意限流与重试，避免 API 超限。

### 阶段 2：算法 + 用户模拟

5. **用户画像提炼**

   - 写脚本：读 `raw/memoir_001.json`，用 LLM 生成 `config/user_persona_memoir_001.json`（格式见 4.3.1）。
   - 若回忆录较长，可先按「全文」生成一个总画像，再按「段」生成段级要点（可选）。
6. **仿真对话流水线**

   - 对 `text2image/` 下每张图：
     - 调用 `MultimodalAnalyzer.analyze_image` → 写 `analysis.json`。
     - 调用 `DialogueManager.start_dialogue` 得到初始问题。
     - 用 **User Sim LLM** 根据画像 + 当前问题 + 当前分析生成回答；再 `add_answer` 得到下一问，循环至设定轮数。
     - 将 `get_dialogue_summary()` 的 `qa_history` 写入 `dialogue.json`。
   - 可封装为单图函数 `run_single_image(image_path, segment_text, persona_path) -> dialogue`，再在批量脚本中遍历所有图调用。
7. **与现有 EvaluationAgent 的关系**

   - 当前 `evaluation_agent.py` 里是模板式回答与简单规则评分。
   - 建议：保留原有逻辑用于快速联调；**用户模拟**部分单独做成「User Sim LLM」模块（新文件或新类），输入画像 + 问题 + 上下文，输出一条回答；评估流水线中「用户回答」统一由该模块生成。
   - 若希望复用 `EvaluationAgent` 的接口，可让 `EvaluationAgent` 支持「从 persona_file 加载画像 + 调用 LLM 生成 answer」，逐步替换模板逻辑。

### 阶段 3：Judge 与评估指标

8. **实现 Judge LLM**

   - 输入：该段的回忆录原文 + 该段对应所有仿真对话拼接成的文本。
   - Prompt：明确「真实回忆录 vs 仿真对话」、维度定义、1–5 分说明、输出 JSON 格式。
   - 输出：写入 `evaluation/memoir_001/seg_01_image_0_judge.json`（或按你们命名规则）。
9. **汇总与报告**

   - 写脚本：遍历 `evaluation/` 下所有 judge 结果，汇总为表格（memoir_id, segment_id, image_id, 内容分, 风格分, 整体评语）。
   - 可算平均分、按段/按图分布，便于写报告和对比不同算法设置。

### 阶段 4：迭代与扩展（可选）

10. **多回忆录 / 多模型对比**

    - 增加更多 `raw/memoir_002.json`，重复 3–9。
    - 若更换问题生成模型或 User Sim 模型，仅重跑 6 和 8，对比同一批图片的 Judge 分数。
11. **人工抽检**

    - 对部分 Judge 结果做人工复核，校准 Judge 的评分标准或 Prompt，使「风格/内容相似度」更符合你们对「好回忆录访谈」的定义。

## 六、与现有代码的衔接小结

| 现有模块                                    | 在仿真测试集中的用法                                                                                                   |
| ------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------- |
| `MultimodalAnalyzer`                      | 直接用于对「仿真图片」做 `analyze_image`，输出写入 `analysis.json`。                                               |
| `DialogueManager` + `QuestionGenerator` | 用分析结果 `start_dialogue` 与 `add_answer`，其中 answer 来自 User Sim LLM。                                       |
| `EvaluationAgent`                         | 可扩展为「从回忆录画像 + LLM 生成回答」，或单独实现 User Sim 模块再在流水线中调用。                                    |
| `server.py` / `main.py`                 | 不需改业务逻辑；新增「仿真数据集生成」脚本，在脚本内实例化 `PhotoStorySystem` 或直接调 analyzer + dialogue_manager。 |
